Let your AI agents talk to each other.

agent-bridge lets a Claude session on one Mac push messages straight into a Claude session on another — agent-to-agent, not shell-to-shell. Installs as a single Claude Code plugin that bundles the MCP tools and the incoming channel. Also works under Codex, Gemini CLI, OpenClaw, and Aider as a plain MCP server.

Why

Agent-to-agent, not shell-to-shell.

Push, not poll

Messages from a remote machine arrive as real-time <channel source="agent-bridge"> events inside your running Claude session. No timers, no busy loops, no stale inbox.

Running sessions, not new ones

The MCP server connects already-running agent sessions. You’re not spawning fresh processes on the remote box — you’re piping into the Claude (or Codex, OpenClaw, Gemini) that’s already sitting there.

Works under every major harness

Full push support under Claude Code and OpenClaw. MCP-tools support under Codex, Gemini CLI, Aider, or anything that speaks the Model Context Protocol.

Hardened against zombies

As of v2.3.2 the MCP server exits cleanly on parent death, handles EPIPE, and expands remote tildes correctly. No more shadow ~/.agent-bridge/inbox directories on the wrong side of the wire.

Install

Pick the harness you use.

Claude Code

Recommended · full push

Installs as a single plugin that bundles the bridge_* MCP tools and the incoming channel. One install, both halves.

Step 1 — clone & build the MCP server once 
# Clone the repo and build the bundled MCP server
git clone https://github.com/EthanSK/agent-bridge.git ~/Projects/agent-bridge
cd ~/Projects/agent-bridge/mcp-server && npm install && npm run build
Step 2 — add the marketplace & install 
claude plugin marketplace add EthanSK/agent-bridge
claude plugin install agent-bridge@agent-bridge
Step 3 — launch alias (both halves active) 
alias claude-tel='claude --dangerously-skip-permissions \
  --channels plugin:telegram@claude-plugins-official \
  --dangerously-load-development-channels plugin:agent-bridge@agent-bridge'
The dev-channel flag is still required. Because a GitHub / local-directory marketplace is a dev source, Claude Code’s channel allowlist rejects it without --dangerously-load-development-channels. The flag takes a plugin:<name>@<marketplace> tagged argument and does two jobs at once: activates the channel and marks it as allowlist-exempt. Don’t also pass --channels plugin:agent-bridge@agent-bridge on top — that creates a duplicate non-dev entry and the allowlist blocks it.

Codex · Gemini CLI · OpenClaw · Aider

MCP server

Any harness that speaks MCP can call the bridge_* tools. Point it at the server you built above.

.mcp.json (or your harness’s equivalent) 
{
  "mcpServers": {
    "agent-bridge": {
      "command": "node",
      "args": ["~/Projects/agent-bridge/mcp-server/build/index.js"]
    }
  }
}

OpenClaw users: the repo also ships an openclaw-plugin/ companion that adds push-style delivery (messages arrive as a fresh user turn) on the OpenClaw side. See the README.

Bash CLI

Non-plugin use

The agent-bridge shell CLI handles pairing, reachability checks, and plain remote-shell diagnostics. Talking to the running agent on another machine happens through the channel plugin's bridge_send_message MCP tool, not the CLI. 

# install.sh puts the bash CLI on your PATH
curl -fsSL https://raw.githubusercontent.com/EthanSK/agent-bridge/main/install.sh | bash

# Then, on each machine
agent-bridge setup
agent-bridge pair

# Plain remote shell on a paired machine (diagnostics only)
agent-bridge run MacBook-Pro "cd ~/Projects/myapp && git status"
Pair two machines

Three steps. Then it’s invisible.

  1. 1

    Setup

    On each machine, run agent-bridge setup. It enables SSH if needed, generates an ED25519 key pair under ~/.agent-bridge/keys/, and creates a one-time pairing token.

  2. 2

    Pair

    Run agent-bridge pair on one machine. It shows a pairing screen with the machine name, IPs, port, and token. Photograph it and send the image to Claude on the other machine — the skill reads the photo and completes the pairing.

  3. 3

    Use

    Once paired, either side can send the other messages via bridge_send_message from inside an agent session — the message is pushed into the running remote agent's context as a <channel source="agent-bridge"> event. Both peers are equal — no fixed client/server roles, and no fresh agent is ever spawned.

How the channel works

Push into a running session.

The plugin registers a single MCP server that advertises the experimental claude/channel capability. A file watcher on ~/.agent-bridge/inbox/ detects new message files and emits notifications/claude/channel into the Claude session — no polling, no manual bridge_receive_messages call needed.

Inside your Claude Code session 
// Machine A's Claude calls:
bridge_send_message({
  machine: "MacBook-Pro",
  content: "can you run the e2e suite and paste the failures?"
})

// A few hundred ms later, Machine B's Claude sees:
<channel source="agent-bridge" from="Mac-Mini" message_id="msg-91c..." ts="2026-04-14T22:11:03Z">
  can you run the e2e suite and paste the failures?
</channel>

// ...and responds the same way:
bridge_send_message({ machine: "Mac-Mini", content: "3 failed, logs attached: ..." })
Tool reference

Seven MCP tools. One concept per tool.

bridge_send_message

Deliver a message to a running agent on a paired machine. Writes a signed message file to the remote inbox over SSH.

bridge_receive_messages

Manually drain the local inbox. Rarely needed under Claude Code — the channel pushes for you — but useful under Codex / Aider.

bridge_list_machines

Show every paired machine with its connection details, key path, and last-seen status.

bridge_status

Probe a specific machine over SSH and report reachability. Uses a short timeout; safe to call on hot paths.

bridge_run_command

Execute a raw shell command on a remote machine and return stdout / stderr / exit code. Use when you want a shell, not an agent.

bridge_clear_inbox

Nuke every message in the local inbox. Handy if you want a clean slate between sessions or tests.

bridge_inbox_stats

Report inbox size, oldest / newest message, watcher health, and last push timestamp. Useful for diagnosing “did my message arrive?”

Security

SSH-authenticated. TTL-bounded. No cloud.

ED25519 keypairs per machine

Each machine generates its own key under ~/.agent-bridge/keys/ during agent-bridge setup. Authorized keys are installed only on paired peers. No passwords, no shared secrets.

Messages signed by the source machine

Every message file carries its origin machine name and a signature. The inbox watcher rejects anything that doesn’t match a known paired peer.

TTL-bounded delivery

Messages have a default 1-hour TTL. Expired files are auto-pruned by the watcher so the inbox never grows unbounded.

Never touches a third-party server

Everything goes directly machine-to-machine over SSH on the ports you already allow. No broker, no relay, no vendor.
