# 4.0.0 — channel plugin rewrite (agent-driven reply routing)

## Context

`openclaw-channel` v3.0.0 dropped the `replyVia` config field and switched
to **agent-driven reply routing**: inbound bridge messages are now surfaced
to the OpenClaw session as a `<channel>` block plus a `[BRIDGE-CONTEXT]`
hint, and the agent decides which reply tools to call (the implicit
`bridge_send_message` leg + any `additionalReplyChannels` user-facing
legs). The plugin no longer mechanically fans out replies. Same model as
the Claude Code channel plugin — both harnesses now behave identically.

The old `replyVia` field is now ignored; configs that still set it will
log a deprecation warning at plugin load time but otherwise work.

This migration doc is the **example migration** referenced in
`docs/migrations/README.md` — it shows the structure the auto-update
probe extracts and injects into `[BRIDGE-UPDATE-AVAILABLE]` bridge
messages so receiving agents know what to do beyond mechanical pull+build.

## Instructions for the agent receiving this update

After the standard `git pull && npm run build && /reload-plugins`:

1. **Audit `~/.openclaw/openclaw.json`** for any
   `channels["agent-bridge"].config.replyVia` or per-target
   `targets.<name>.replyVia` keys. Remove them — they are no longer
   read by the plugin and only generate deprecation warnings on every
   plugin load.
2. **Verify the plugin loaded the v3+ channel** by tailing
   `~/.openclaw/logs/gateway.log` for the line
   `Agent Bridge (Channel v3 — agent-driven routing)`. If you still see
   `v2.x` or `replyVia` warnings, the runtime is still on the old build —
   restart the OpenClaw gateway (`launchctl kickstart -k
   gui/$(id -u)/ai.openclaw.gateway`) and re-check.
3. **If `additionalReplyChannels` is unset on a target whose
   `openclaw_channel` is `"telegram"`,** the plugin defaults to
   `["telegram"]` so existing setups keep their Telegram fan-out. No
   action needed — just be aware the agent is expected to call the
   user-facing reply tool itself now.
4. **Confirm `bridge_send_message` is available** in the OC agent's tool
   list. If not, the openclaw plugin entry in
   `~/.openclaw/openclaw.json` is missing or stale — re-run
   `agent-bridge plugin-registry-rewire` to fix.

## Verification

- `tail -50 ~/.openclaw/logs/gateway.log | grep -i "agent-bridge"` shows
  `Channel v3 — agent-driven routing`.
- Sending a test bridge message round-trip (e.g.
  `bridge_send_message machine=local target=openclaw/default
  message="ping"`) surfaces a `<channel source="agent-bridge">` block
  followed by a `[BRIDGE-CONTEXT]` block in the OC session.
- No `replyVia` deprecation warnings in the gateway log.

## Rollback

Pin to `openclaw-channel@2.4.x` by checking out the pre-3.0.0 commit in
the workspace clone and running `npm install` — but you'll lose Claude
Code parity and have to re-apply the `replyVia` config. Forward is the
expected path.
