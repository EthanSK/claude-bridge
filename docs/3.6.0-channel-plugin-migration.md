# 3.6.0 — `claude-code-channel` plugin migration

**Status:** design  ·  **Author:** Claude (Mac-Mini, design pass)  ·  **Date:** 2026-04-21

> **TL;DR.** Move the `claude-code` channel role out of the stdio MCP server
> and into a dedicated Claude Code plugin (peer of `openclaw-channel/`).
> Demote `mcp-server` to a tools-only host. Adopt the five Telegram-plugin
> patches verbatim. Wire format on disk is unchanged so 3.5.x ↔ 3.6.x peers
> interoperate during rollout.

---

## 1 · Why

### 1.1 The bug 3.6.0 actually solves

`mcp-server`'s current `channel-owner` role is structurally fragile because
the MCP server is a **stdio child of a single Claude tool turn**. It owns
three things at once:

1. The `bridge_*` MCP tools (per-turn JSON-RPC).
2. The polling inbox watcher for `~/.agent-bridge/inbox/claude-code/`.
3. The `notifications/claude/channel` push pipe back to the running session.

The MCP child's lifetime is governed by Claude Code's MCP host, which is
allowed to kill, restart, or re-pipe the stdio child between tool turns —
and does. The 3.5.x patch series (heartbeat, sibling-MCP step-down, SIGPIPE
swallow, watcher lease, broken-pipe diagnostics, "obey Claude MCP lifecycle
shutdown") all attempt to **work around** the lifetime mismatch by keeping
the watcher alive across pipe events the host treats as "you're done now".
Each patch closes one race; the next one opens. We are now four patch
levels deep without a quiescent state.

The structural fix is: **stop using an MCP stdio child to host a session-
lifetime channel.** Channel ownership belongs in a process whose lifetime
matches the Claude session, not the tool turn.

### 1.2 Telegram is the existence proof

The Telegram channel plugin
(`~/.claude/plugins/cache/claude-plugins-official/telegram/0.0.6/`) is
declared in its plugin manifest as an MCP-style server (`bun run start`
spawning `server.ts`) but Claude Code's plugin host launches it **once per
session** rather than per tool turn, gives it a stable stdin/stdout pair,
and only tears it down when the session ends. The mini-Mac instance pid
7511 has been alive `3d 17h` at the time of writing, with the heartbeat
line marching once per minute in `~/.claude/channels/telegram/server.log`:

```
[heartbeat] uptime=320789s ppid=12345 rss=82MB
[heartbeat] uptime=320849s ppid=12345 rss=82MB
…
```

`agent-bridge` should ship the same shape: a Claude Code plugin whose
`server.ts` (or `index.ts`) is exactly the channel host. The MCP tools
then become a *separate* stdio MCP child — which Claude Code is free to
respawn across turns without affecting the watcher.

### 1.3 What stays the same

* The on-disk wire format (`BridgeMessage` JSON, the per-target inbox
  layout `~/.agent-bridge/inbox/<target>/*.json`, the `.archive` /
  `.failed` subdirs, the `.delivered` / `.processed` ledgers).
* The watcher lease file (`~/.agent-bridge/locks/claude-code.watcher-lock.json`).
* The CLI (`agent-bridge pair`, `status`, `list`, `run`, `connect`).
* The `bridge_*` MCP tool surface (callers who use the tools see no change).
* Same `~/.agent-bridge/keys/` SSH keys for outbound SCP.

3.5.x peers sending into a 3.6.x receiver work unchanged, and vice versa.

### 2.2 Wire-format compatibility statement

3.5.x and 3.6.x peers fully interoperate during rollout because:

* `BridgeMessage` JSON shape is unchanged (`id`, `from`, `to`, `type`,
  `content`, `timestamp`, `replyTo`, `ttl`, `target`, `fromTarget`).
* Inbox subdir convention (`inbox/claude-code/`, `inbox/openclaw/<acct>/`)
  is unchanged.
* The watcher lease file format and path are unchanged — so a 3.5.x
  `mcp-server` running in `channel-owner` mode and a 3.6.x
  `claude-code-channel` plugin will mutually back off via the existing
  lease (only one wins at a time, the other steps down to standby /
  tools-only).
* The CLI's outbound SCP path (`scp <pkg> remote:~/.agent-bridge/inbox/<target>/<id>.json`)
  doesn't change.

This means rollout can be **per-machine**: upgrade Mac-Mini to 3.6.x while
MBP stays on 3.5.x; messages flow both ways throughout.

### 2.3 Plugin manifest (how Claude Code spawns it)

`claude-code-channel/.claude-plugin/plugin.json`:

```jsonc
{
  "name": "agent-bridge",
  "description": "Cross-machine agent-to-agent channel for Claude Code. Pushes <channel source=\"agent-bridge\"> messages from paired machines into the running session, and exposes bridge_* MCP tools for sending.",
  "version": "3.6.0",
  "keywords": ["agent-bridge", "channel", "mcp", "ssh", "agent-to-agent", "cross-machine", "claude-code"],
  "author": { "name": "EthanSK", "url": "https://github.com/EthanSK/agent-bridge" }
}
```

`claude-code-channel/.mcp.json` (matches Telegram's bun-run pattern):

```jsonc
{
  "mcpServers": {
    "agent-bridge": {
      "command": "bun",
      "args": ["run", "--cwd", "${CLAUDE_PLUGIN_ROOT}", "--shell=bun", "--silent", "start"]
    }
  }
}
```

`claude-code-channel/package.json` `start` script:

```jsonc
{
  "scripts": {
    "start": "bun install --no-summary && bun src/index.ts"
  }
}
```

The bun-run-from-source pattern (no precompile step) is what gives
Telegram its multi-day lifetime: the script *is* the long-running
process and Claude Code's plugin host treats it as a session-scoped
child. We adopt the same pattern verbatim.

The existing `mcp-server/.mcp.json` is *removed* from the marketplace
plugin entry as the channel host. It can either:

* be split into a separate **`agent-bridge-tools`** plugin (recommended),
  whose `.mcp.json` runs `node build/index.js` and exposes the `bridge_*`
  tools without any watcher,  *or*
* be folded into `claude-code-channel/.mcp.json` as a second `mcpServers`
  entry. (Probably not — keeping them separate makes lifetimes
  unambiguous.)

### 2.4 Lifecycle — what the user sees

| Event                                  | 3.5.x behaviour                                                                                            | 3.6.x behaviour                                                                                                       |
| -------------------------------------- | ---------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------- |
| Claude Code session starts             | `mcp-server` stdio child spawns; auto-elects watcher lease if `AGENT_BRIDGE_ROLE=channel-owner`.           | `claude-code-channel` plugin spawns once via `bun run`; takes the watcher lease for the whole session.                |
| Tool turn invokes `bridge_send_message`| Same `mcp-server` child handles it.                                                                        | A *separate* `agent-bridge-tools` MCP stdio child handles it (or the same plugin, see §2.3 footnote). The watcher is unaffected. |
| Tool turn ends                         | MCP host may close stdio; 3.5.x patches A–F try to keep the watcher alive.                                 | MCP host can freely respawn the tools child; watcher in the plugin process is undisturbed.                            |
| Inbound `<channel source="agent-bridge">` push | Polling watcher inside `mcp-server` writes `notifications/claude/channel`.                                  | Polling watcher inside `claude-code-channel` writes `notifications/claude/channel`. Path identical to Telegram's push. |
| Session ends (Claude Code closes)      | stdio EOF → cascading shutdown attempts → fatalTransportExit, lease released.                              | stdin EOF → `shutdownWithReason('stdin-end')` (Patch C) → orphan watchdog confirms (Patch A) → graceful release.      |
| User runs `claude mcp list`            | Sees `agent-bridge` (channel-owner).                                                                       | Sees `agent-bridge` (channel) AND optionally `agent-bridge-tools` (tools host).                                       |

## 3 · How (architecture)

### 3.1 Module boundaries — what moves where

| File (source of truth in 3.5.x)          | 3.6.x destination                                              | Notes                                                                                       |
| ---------------------------------------- | -------------------------------------------------------------- | ------------------------------------------------------------------------------------------- |
| `mcp-server/src/watcher.ts`              | **moves** → `claude-code-channel/src/watcher.ts`               | Polling loop, lease, archive/failed handling, channel-notify callback.                      |
| `mcp-server/src/inbox.ts`                | **moves** → `claude-code-channel/src/inbox.ts`                 | Delivered/processed bookkeeping, replay-undelivered, in-memory cache.                       |
| `mcp-server/src/index.ts` (channel push) | **factors out** → `claude-code-channel/src/notify.ts`          | `notifications/claude/channel` JSON-RPC writer. Trimmed of EPIPE-survival hacks (Patch C handles it now). |
| `mcp-server/src/config.ts`               | **shared subset** → `claude-code-channel/src/config.ts`        | Both packages need `BRIDGE_DIR`, `INBOX_DIR`, `LOCKS_DIR`, `inboxSubdir`, `isValidTarget`. Either copy/paste (zero-dep philosophy) or extract a tiny `@agent-bridge/shared-config` workspace package — see §5 open questions. |
| `mcp-server/src/log.ts`                  | **shared / copy** → `claude-code-channel/src/log.ts`           | NDJSON appender to `~/.agent-bridge/logs/agent-bridge.log` is identical.                    |
| `mcp-server/src/logger.ts`               | **stays in mcp-server**                                        | Per-component verbose log, used by tools.                                                   |
| `mcp-server/src/index.ts` (tools host)   | **stays in mcp-server**, slimmed                               | Pure MCP transport + `registerTools(server)`. No watcher, no notify, no lease.              |
| `mcp-server/src/tools.ts`                | **stays in mcp-server**                                        | `bridge_send_message` etc. unchanged.                                                       |
| `mcp-server/src/ssh.ts`                  | **stays in mcp-server**                                        | Used by tools.                                                                              |
| `mcp-server/src/pathCache.ts`            | **stays in mcp-server**                                        | Used by tools.                                                                              |
| `mcp-server/src/index.ts` (Patch A–F)    | **moves & restructures** → `claude-code-channel/src/lifecycle.ts` | The lifecycle patches belong with the long-lived process. The tools child no longer needs them. |

After Phase 4, `mcp-server/src/` is roughly **half its current size**
(currently 7 files, ~3,900 LOC; post-migration ~5 files, ~1,700 LOC).

### 3.2 The five Telegram patches and where each lands

All five patches live in `claude-code-channel/src/lifecycle.ts` (or are
inlined at the top of `src/index.ts` mirroring Telegram's `server.ts`
layout). The 3.5.x analogues in `mcp-server/src/index.ts` are deleted as
part of Phase 4.

| Patch | Telegram name                                  | What it does                                                                                       | Lands where                                                                                                                                                                                                                                                                  |
| ----- | ---------------------------------------------- | -------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **B** | "stderr tee to local log"                       | Mirror `process.stderr.write(...)` into `~/.agent-bridge/logs/claude-code-channel.log`, rotate at 5 MB → keep last 2 MB. Runs **before** any other stderr write. | Top of `src/index.ts`, identical to Telegram's lines 31–51. Log path: `~/.agent-bridge/logs/claude-code-channel.log` (deliberately separate from the unified NDJSON log so post-mortem diagnostics don't fight with the structured event stream). |
| **C** | `shutdownWithReason(reason, detail)`            | Single funnel for every teardown trigger; logs the reason to stderr (and via Patch B, to file).    | `src/lifecycle.ts`. Hook `process.stdin.on('end' \| 'close')`, `process.on('SIGTERM' \| 'SIGINT' \| 'SIGHUP')` to it. Internal `shutdown()` performs handle dump (Patch E) + lease release + `setTimeout(process.exit, 2000)` + `setTimeout(SIGKILL, 5000).unref()` backstop. |
| **D** | "60s heartbeat (refed for channel-owner only)" | `setInterval(() => stderr.write('[heartbeat] uptime=Xs ppid=Y rss=ZMB\n'), 60_000).unref()`. Through Patch B this lands in `claude-code-channel.log`. | `src/lifecycle.ts`. **Refed for the channel-owner process**, `unref`'d for any tools-only mcp-server processes that retain a heartbeat for diagnostics. The channel plugin always wants the event loop alive; the tools host doesn't. |
| **E** | "shutdown handle/request dump"                 | At shutdown entry, `process._getActiveHandles()` + `process._getActiveRequests()` counts dumped to stderr alongside RSS. Reveals what kept the loop alive at teardown (e.g. an outstanding fs watch). | Inside the `shutdown()` body in `src/lifecycle.ts`. Telegram's lines 671–681.                                                                                                                                                                |
| **A** | "3-poll confirmation orphan watchdog"          | Every 5 s, check `process.ppid !== bootPpid \|\| process.stdin.destroyed \|\| process.stdin.readableEnded`. After **3 consecutive** orphaned polls (15 s), call `shutdownWithReason('orphan-watchdog', detail)`. Any clean poll resets the counter. | `src/lifecycle.ts`. Telegram's lines 718–737. Critical for the bun-run-from-script pattern: stdin events alone don't reliably fire when `bun run` wrapper → shell → us is severed; the watchdog covers that case.                            |

(There is also Telegram's **Patch F** — heartbeat-recency guard against
parallel subagent spawns murdering the parent poller. We adopt this one
too; it lives at the top of `src/index.ts` and uses the watcher lease's
heartbeat field instead of mtime on `server.log`. Documented in §3.4.)

### 3.3 Lifecycle flow diagram

```
┌─────────────────────────────────────────────────────────────────────┐
│ Claude Code session starts                                           │
└──┬──────────────────────────────────────────────────────────────────┘
   │
   ├── plugin host launches: bun run claude-code-channel/src/index.ts ─┐
   │                                                                   │
   │   Patch B: tee stderr to claude-code-channel.log                  │
   │   Patch F: heartbeat-recency guard against parallel spawn         │
   │   Take watcher lease (fail-soft if held by a healthy peer)        │
   │   Start polling watcher → notifications/claude/channel push       │
   │   Patch D: 60s heartbeat begins                                   │
   │   Patch A: orphan-watchdog 5s polling begins                      │
   │   Patch C: shutdown funnel installed                              │
   │                                                                   │
   │   ╔══════════════════════════════════════════════════╗            │
   │   ║ Process stays alive for the entire session       ║            │
   │   ║ (matches Telegram's pid-7511-for-3-days shape)   ║            │
   │   ╚══════════════════════════════════════════════════╝            │
   │                                                                   │
   │   Session end → stdin EOF → shutdownWithReason('stdin-end')       │
   │                  Patch E dumps handles → release lease → exit     │
   │                                                                   │
   └─────────────────────────────────────────────────────────────────► fin
                                                                       
   ──────────────────────── orthogonally ─────────────────────────
                                                                       
   ├── Tool turn begins → MCP host spawns: node mcp-server/build/index.js
   │     (or reuses an existing one)                                   │
   │     Tools-only mode: watcher disabled, no lease attempt           │
   │     Handles bridge_send_message / bridge_run_command / etc.       │
   │                                                                   │
   │   Tool turn ends → MCP host may kill / respawn freely; watcher    │
   │   in claude-code-channel above is unaffected.                     │
   │                                                                   │
   └─────────────────────────────────────────────────────────────────►
```

### 3.4 Coexistence: lease as the coordination point

`claude-code-channel` and any `mcp-server` (3.5.x channel-owner OR 3.6.x
tools-only) running at the same time share **only** the watcher lease
file at `~/.agent-bridge/locks/claude-code.watcher-lock.json`.

* 3.6.x tools-only never touches the lease.
* 3.6.x `claude-code-channel` is the lease holder.
* 3.5.x `mcp-server` channel-owner (running on the *other* machine, or
  briefly during local upgrade) may hold the lease; the new plugin's
  bootstrap respects an existing healthy lease (mtime within 90 s,
  pid alive) and exits 0 — same logic as Telegram's Patch F.

This is the inter-version coexistence story: the lease file IS the
contract, both sides honour it, no version bump is required to roll
out per-machine.

## 4 · Migration plan

### Phase 0 — 3.5.5 lifeboat (in flight, parallel subagent)

Lands the last batch of EPIPE/lifecycle hardening on 3.5.x so anyone who
stays on 3.5.x for a while keeps a usable channel. Out of scope for this
design doc; tracked separately by the parallel subagent editing
`mcp-server/src/*` right now.

**Done when:** `mcp-server` 3.5.5 published with the latest patches.

### Phase 1 — `claude-code-channel/` skeleton (this PR, optional)

* Create `claude-code-channel/{package.json, .claude-plugin/plugin.json, .mcp.json, src/index.ts}`.
* `src/index.ts` is a **stub** — the only "logic" is:
  * The 5 imports it'll need (`@modelcontextprotocol/sdk/server/stdio.js`,
    `node:fs`, `node:os`, `node:path`, `node:process`).
  * A guard `if (process.env.AGENT_BRIDGE_PLUGIN_STUB !== 'activate') process.exit(0);`
    so a stray `bun run` from a developer can't accidentally fight the
    real channel-owner.
  * A `// TODO(3.6.0): port watcher.ts + lease + Patches A–F here` stanza.
* `package.json` has `version: "0.0.0-stub"` so npm publish would refuse.
* Marketplace.json is **NOT** updated yet. The skeleton is filesystem-only.

**Done when:** the skeleton compiles under `bun --check` (or `tsc --noEmit`)
and the inert guard works.

### Phase 2 — port the channel logic

* Copy `mcp-server/src/watcher.ts`, `inbox.ts`, the channel-push helper,
  and the lifecycle patches into `claude-code-channel/src/`.
* Wire `notify.ts` to write `notifications/claude/channel` on the same
  stdout JSON-RPC framing the MCP SDK uses for the plugin's own MCP
  server (or, if we go server-less like Telegram's Patch-F-only flavour,
  build a thin SDK shim).
* Add `claude-code-channel` as a *second* plugin entry in
  `marketplace.json`. Until version is bumped this is dev-only.
* Test side-by-side with `mcp-server` 3.5.x channel-owner: the lease
  arbitration should let exactly one win.

**Done when:** a fresh Claude Code session with the new plugin shows
the same `notifications/claude/channel` push behaviour the 3.5.x
channel-owner showed, *and* its uptime survives 10 successive tool
turns without re-spawn (proxy: `ps -o etime= -p $(cat
~/.agent-bridge/locks/claude-code.watcher-lock.json | jq .pid)`).

### Phase 3 — switch `mcp-server` default role to tools-only

* Change `mcp-server/src/index.ts` so `AGENT_BRIDGE_ROLE` defaults to
  `tools-only` when unset (currently the legacy auto path).
* Keep `AGENT_BRIDGE_ROLE=channel-owner` as an opt-in for hosts that
  haven't installed `claude-code-channel` (Codex, Gemini-CLI, plain
  shell experimentation).
* Bump `mcp-server` to 3.6.0-rc.

**Done when:** new Claude Code installs default to "channel = plugin,
tools = mcp" and the legacy path is reachable only via env override.

### Phase 4 — remove channel-owner code from `mcp-server`

* Delete `mcp-server/src/watcher.ts`, `mcp-server/src/inbox.ts`, the
  channel-push helper, and Patches A–F from `mcp-server/src/index.ts`.
* Remove the `channel-owner` branch entirely from `index.ts`. Remove the
  `AGENT_BRIDGE_ROLE` env var (or keep the var but make `channel-owner`
  return a clear error pointing at `claude-code-channel`).
* Drop `notifications/claude/channel` plumbing.
* Slim `mcp-server/.mcp.json` env block (no more `AGENT_BRIDGE_ROLE`).

**Done when:** `mcp-server/src/` is tools-only; `bridge_inbox_stats`
reflects watcher state by *reading* the lease file rather than owning it.

### Phase 5 — validate + ship 3.6.0

* CHANGELOG entry.
* AGENTS.md / INSTRUCTIONS.md / global CLAUDE.md updates so future
  sessions know channel ≠ tools.
* Soak protocol parity test (see §6.2).
* Tag `v3.6.0`, push, both machines reload-plugins.

## 5 · Risks / open questions

### 5.1 Does Claude Code's plugin host spawn channel plugins independently of MCP children?

**Evidence yes**, from Telegram. Telegram's `.mcp.json` declares an
`mcpServers.telegram` entry with `command: bun, args: ['run', '--cwd',
'${CLAUDE_PLUGIN_ROOT}', '--shell=bun', '--silent', 'start']` — same shape
we propose. Yet pid 7511 has been alive >3 days on the Mini, surviving
hundreds of tool turns. Conclusion: the plugin host spawns the manifested
MCP server **once per session**, not per turn, when the plugin manifests
itself as a channel.

**Confirmation work for Phase 1:** before porting any logic, verify with
a no-op skeleton (Phase 1 above) that pid persists across ≥ 5 tool turns
on a real Claude Code session. If it doesn't, we may need a different
plugin shape (e.g. declaring `channels: ["agent-bridge"]` in plugin.json
the way Telegram has been observed to). Treat Phase 1 as a load-bearing
spike.

### 5.2 Lease handoff during rollout: 3.5.x channel-owner vs 3.6.x channel-plugin

When MBP is 3.5.x and Mini is 3.6.x, no problem — leases are local.

When **the same machine** has both installed transiently (e.g. user
upgrades but old `mcp-server` is still spawned by `claude mcp list` for
older sessions), the lease file mediates. Both implementations check:

```text
if lease file exists AND pid alive AND lease.updatedAt within 90s:
    back off, exit 0
else:
    take lease, continue
```

We can audit both code paths and confirm they share the 90 s freshness
window (3.5.x already has this for sibling-MCP step-down). Add a
`leaseHolderVersion` field to the lease metadata for diagnostics — not
required for correctness.

### 5.3 Does `mcp-server` tools-only mode need to "find" the channel plugin?

**Probably not.** Tools and channel coordinate exclusively through the
filesystem (inbox subdir + lease file). Tools call `bridge_send_message`
→ SCP to remote inbox → remote's plugin watcher picks it up. Tools don't
need to talk to the local plugin process at all.

The one corner case: `bridge_inbox_stats` currently reports watcher
backend, lease state, queue depth. In tools-only mode it must read these
out of the lease file + filesystem (the watcher is in another process).
Implementation: the lease file already carries `pid`, `target`, `role`,
`token`, `startedAt`, `updatedAt`. Add `backend` and `queueDepth` if
needed (queueDepth can also be computed by `readdirSync(inboxSubdir)`
length on demand).

### 5.4 Shared config — copy/paste or workspace package?

Three modules need the same constants/helpers (`BRIDGE_DIR`, `INBOX_DIR`,
`LOCKS_DIR`, `inboxSubdir`, `isValidTarget`, `CLAUDE_CODE_TARGET`,
`PRUNE_MAX_AGE_MS`).

Options:

| Option                                          | Pro                                                | Con                                                                                |
| ----------------------------------------------- | -------------------------------------------------- | ---------------------------------------------------------------------------------- |
| Copy/paste (~80 LOC duplicated)                 | Zero deps, mirrors `openclaw-channel` philosophy.  | Drift risk; small.                                                                 |
| Workspace `@agent-bridge/shared` package        | Single source of truth.                            | Adds a workspace setup; both packages must publish or use file: deps.              |
| Symlink + `tsc --rootDir` trickery              | No package overhead.                               | Brittle, surprising to readers.                                                    |

**Recommendation:** copy/paste for 3.6.0. Re-evaluate at 4.x if drift
becomes painful.

## 6 · Validation plan

### 6.1 Don't-break-3.5.x peers test

1. Mini upgraded to 3.6.x (`claude-code-channel` plugin + tools-only mcp).

## 8 · Implementation log (post-design)

The 3.6.0 implementation landed on 2026-04-21. A few decisions diverged
slightly from the original design above; recording them here for the
post-mortem.

* **Node + tsc, not Bun.** The Telegram plugin uses `bun run` from source.
  We chose `node build/index.js` for the channel plugin to match
  mcp-server's existing toolchain (TypeScript with `tsc`, `@modelcontextprotocol/sdk`,
  no Bun dep). The "session-scoped lifetime" property comes from declaring
  the `experimental.claude/channel` capability in the MCP server, not from
  the runtime — Claude Code's plugin host treats both spawn shapes the same
  way. Verified empirically by lifecycle tests in
  `claude-code-channel/test/lifecycle.test.mjs`.
* **Shared config: copy/paste, not workspace package.** As recommended in
  §5.4 above. `claude-code-channel/src/{config,log,inbox,watcher}.ts` are
  copy-paste subsets of the mcp-server originals. Drift risk is acknowledged
  and re-evaluation deferred to 4.x. The two `config.ts` files MUST stay
  byte-compatible on `LOCKS_DIR`, `CLAUDE_CODE_TARGET`, and `inboxSubdir`
  for lease arbitration to work across mixed-version peers.
* **MCP server name: `agent-bridge-channel`.** The plugin's MCP server name
  (declared inside `new McpServer({ name: ... })`) is `agent-bridge-channel`,
  distinct from the existing `agent-bridge` (tools-only mcp-server). This
  surfaces as `agent-bridge-channel` in `claude mcp list` and prevents name
  collisions between the two stdio children.
* **Patch F uses lease `updatedAt`, not server-log mtime.** Telegram's
  Patch F probes `mtime(server.log)` to decide if an existing pid is
  healthy. We replaced that with the watcher lease's `updatedAt` field,
  which is a more direct signal (the lease owner writes it every 5 s via
  the renew heartbeat). Same intent, slightly cleaner contract, and works
  correctly across mixed 3.5.x↔3.6.x peers since both write the field.
* **Default `AGENT_BRIDGE_ROLE` flipped to `tools-only`.** The
  `mcp-server/.mcp.json` env block now sets `AGENT_BRIDGE_ROLE=tools-only`
  by default. Setting `AGENT_BRIDGE_ROLE=channel-owner` remains supported
  as an opt-in for non-Claude hosts (Codex, Gemini-CLI). Matches Phase 3
  of the migration plan.
